# AI Visibility Assessment
Purpose: Provide an assessment to the user, the owner or manager of a food establishment, on their business visibility torwards AI assistants like ChatGPT, Claude, Gemini AI, and Perplexity based on how well they align with best practices for maximum AI visibility.

## 1. Problem Statement
Food establishments need to provide well structured data for AI assistants in order to be considered when diners use an AI assistant to search for a restaurant by name, by cuisine, by dish, or dietary preferences. 

Most food establishments have websites without proper data, with PDF menus, and menus with different prices across different platforms like Google Business Profile, Yelp, DoorDash, or Uber Eats. 

## 2. Goals and Non-Goals
### 2.1 Goals
- Build the capability to identify the right business by reusing the logic from free AI Visibility Report (finding the business using Google Business Profiles)
- Create a benchmark of best practices for maximum AI visibility for food establishments including business info and menu
- Build the capability to run an assessment for an specific business that provides a score from 0 to 10 comparing what the business offers with the benchmark
- Build the capability to provide recommendations on what to do to improve the score, starting with quick wins, based on the result of the assessment
### 2.2 Non-Goals
- The assessment will not run prompts with AI assistants, simulating users to see responses.
## 3. System Overview
The report is generated by Synvya and published to Nostr using encrypted seal wrapped events with the sender being Synvya and the receiver being the food establishment business. 

The Synvya nostr public and private keys are stored in AWS Secrets Manager as a single secret with two keys: `npub` and `nsec`.

The system should use the following relays: `wss://relay.damus.io` and `wss://relay.snort.social`.

### 3.1 Main Components
1. UX
- The UX resides in the client
- The UX includes a button to run an assessment
- The UX provides displays the assesment of the last run
- The UX retrieves the last assessment from Nostr
2. Assessment Engine
- The AI Visibility Asessment is executed as a lambda function in AWS. 
- The user triggers the execution of the AI Visibility Assessment in the client.
- The client sends to the lambda function the information required to run the assessment including at least:
  - The Nostr public key of the business
  - The Google Place ID of the business
3. Assesment report storage
- The report is stored as a Nostr event `kind 30078` with the following format:
```yaml
  {
    "id": "<usual hash>",
    "pubkey": "<synvya-public-key>",
    "created_at": "<current-time>",
    "kind": 30078,
    "tags": [
      ["d", "synvya-visibility-assessment"],
      ["p", "<receiver-pubkey>"],
    ],
    "content": "ai-visibility-assessment-report-content",
  }
```
- The Nostr event `kind 30078` is not published to the Nostr network. Instead, the event is sealed (`kind:13`) and then gift-wrapped (`kind:1059`) following the same procedure as NIP-17 with the receiver being the food establishment. The sealing and gift wrapping ensure that external observers do not know that the food establishment is a customer of Synvya and can't access the AI Visibility Assessment.

## 5. Legacy Code Improvements

The Google Place ID should be stored in the `kind 0` event of the business with the tag `["google_places_id", "<google-place-id>"]`. When the profile is reloaded from Nostr, if the `kind 0` event contains the Google Place ID, then there is no need to confirm it again.
